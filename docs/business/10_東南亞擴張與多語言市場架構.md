# 10｜東南亞擴張與多語言市場架構

> 這份文件給誰看：想把 POS Easy 從印尼擴到整個東南亞的創業者，以及負責實作多語言/多市場架構的工程 agent
> 讀完你能做到：理解「語言≠市場」的解耦設計、五國場景公約數、三環漸進式賺錢模式與擴張順序
> 前置文件：01_商業模式選擇.md（版本 D／L）、docs/GO_TO_MARKET.md

## 1. 核心心法：語言 ≠ 市場（兩個獨立開關）

最容易犯的錯是把「翻譯成泰文」當成「進入泰國」。實際上是兩層：

| 開關 | 管什麼 | 誰在切 |
|---|---|---|
| **語言（Language）** | 介面顯示文字。現有 id/en/zh，未來加 vi/th | 使用者隨時切，純顯示 |
| **市場（Market Profile）** | 貨幣與格式、國家 QR 支付標準、主流通訊軟體、賒帳文化稱呼、手機號格式、定價表、種子資料 | 開店設定時選一次，之後幾乎不動 |

兩者**預設配對但獨立**：泰國店預設泰文＋泰銖，但曼谷的華人老闆可以切中文介面、市場仍是泰國；印尼店的介面切英文，錢照樣顯示 Rp。這樣加一個新語言不會牽動任何商業邏輯，加一個新市場也不用等翻譯全好。

## 2. 技術架構（刻意簡單，一次到位）

**已經打好的地基（Phase 1 的紅利）**：全 App 金額都走 `fmtMoney()`、文字都走 `t()`——所以多市場化只要改兩個中心點，不用動 21 個頁面。

```
src/market/profiles.js   ← 新增：一國一個 profile 物件（各約 50 行）
{
  id: { currency:'IDR', symbol:'Rp',  thousandsSep:'.', decimals:0,
        nationalQR:'QRIS',     chatApp:'WhatsApp',  phonePrefix:'62',
        creditLedgerName:'Kasbon',  priceTable:{warung:49000, resto:129000} },
  th: { currency:'THB', symbol:'฿', decimals:0,
        nationalQR:'PromptPay', chatApp:'LINE',     phonePrefix:'66',
        creditLedgerName:'เงินเชื่อ', priceTable:{warung:139, resto:349} },
  vn: { currency:'VND', symbol:'₫', ...  nationalQR:'VietQR', chatApp:'Zalo', ... },
  my: { currency:'MYR', symbol:'RM', decimals:2, nationalQR:'DuitNow QR', chatApp:'WhatsApp',
        creditLedgerName:'Hutang', ... },
  ph: { currency:'PHP', symbol:'₱', ...  nationalQR:'QR Ph', chatApp:'Messenger',
        creditLedgerName:'Utang (Listahan)', ... },
}
```

實作要點（工程 agent 照此執行）：
1. `fmtMoney()` 改讀取當前 market profile（取代硬編 Rp）——因為全 App 已走此函數，**零頁面改動**
2. localStorage 兩把獨立鑰匙：`pos_language`（既有）＋ `pos_market`（新增），互不牽動
3. i18n 沿用現有 fragment 架構：每個 fragment 加 `vi`/`th` 區塊即可，parity 測試自動涵蓋新語言（一鍵抓漏譯）
4. Supabase `licenses` 表加 `market` 欄位＋`reseller_id` 欄位（見 §4 環 3）——定價表放 profile，你在後台調價**不用發版**
5. 手機驗證、日期格式、收據格式全部從 profile 取

**刻意不做的（防複雜化）**：
- ❌ 不做 IP 自動偵測市場（誤判率高、老闆會困惑）——開店設定選一次就好
- ❌ 不做逐功能翻譯開關——語言是整包切
- ❌ 不把市場切換放進日常 UI——藏在設定頁，因為它一年切不到一次
- ❌ 不為任何國家 fork codebase——永遠一份代碼＋profiles

## 3. 為什麼這套系統天生適合全東南亞（五國場景公約數）

| 國家 | 小店俗稱 | 賒帳文化 | 國家 QR 標準 | 生意用通訊軟體 | 貨幣 |
|---|---|---|---|---|---|
| 印尼 | Warung | Kasbon | QRIS | WhatsApp | Rp |
| 馬來西亞 | Kedai runcit | Hutang | DuitNow QR | WhatsApp | RM |
| 菲律賓 | Sari-sari store | Utang（listahan 記帳本） | QR Ph | Messenger | ₱ |
| 越南 | Tiệm tạp hóa | Ghi nợ | VietQR | Zalo | ₫ |
| 泰國 | โชห่วย（Chohuay） | เงินเชื่อ | PromptPay | LINE | ฿ |

五國**同構**：現金為主＋國家統一 QR 支付、街坊賒帳記在本子上、用通訊軟體做生意、家庭式小店無數位工具。這代表產品核心（收銀／庫存／賒帳／報表／QR 點餐）**100% 可移植**，每進一國只需要換 profile 層＋語言包＋在地話術。Kasbon（任務 A8）開發時就要按「賒帳模組」設計而不是「Kasbon 專屬功能」——名稱從 profile 讀，一次開發五國通用。

## 4. 漸進式賺錢模式：三環擴張

**環 1｜印尼直營（現在 → 500 店）**
按 01/GTM 既定路線：混合制訂閱＋服務費＋限量終身版，之後疊硬體與市集。印尼是東南亞最大市場（人口與小微企業數量），你已投入、文件與素材齊全——**擴張前先在這裡把模式跑通並存下現金流**。這一環的另一個隱形產出：被驗證過的 playbook（10 份手冊）本身變成環 3 要賣的資產。

**環 2｜產品化「國家包」（印尼 500 店後）**
把「進入一個國家」封裝成標準交付物：`語言字典 + market profile + 在地支付/通訊對接說明 + 翻譯過的銷售手冊（03/06/07）`。目的：把開新國家的成本從「重做一遍」降到「填一張表」。這一環不賺錢，是把環 3 的貨架備好。

**環 3｜國家授權（版本 L 落地，賺授權費＋分成）**
越南、泰國**不要自己營運**——語言、在地信任、客服都是你的劣勢。把國家經營權授權給在地運營者：
- 收費：授權費 US$5.000–15.000/國 ＋ 營收分成 5–10%（與 01 §Part 2 版本 L 一致）
- 他負責：在地行銷、客服、收款、代理網絡（等於他在他的國家重演你的 07 手冊）
- 你負責：產品、雲端、控制面板
- **掌控權不變**：所有終端店家的 license 都在你的 Supabase，國家運營者只是 `licenses.reseller_id` 的一層；違約＝切權，客戶資料與系統仍在你手上。這是版本 D 代理制的國家級放大，同一套技術底座。

**定價按購買力在地化**（放 profile，後台可調）：ID Rp 49.000 ≈ US$3 為基準 → MY ~RM 19、TH ~฿139、PH ~₱179、VN ~₫79.000（進場時以當地同類產品價格校準）。

## 5. 擴張順序建議（一次一國，吃透再下一國）

1. **印尼**（現在）——主場，直營
2. **馬來西亞**——過渡成本最低：WhatsApp 相同、英語/馬來語通（馬來語與印尼語高度互通，翻譯成本近零）、購買力高；可先直營試水
3. **菲律賓**——英語市場（現有 en 語言包直接用！）、utang 文化全東南亞最強、Messenger 對接
4. **越南／泰國**——最後做：必須靠環 3 在地運營者（Zalo/LINE 生態、語言門檻），屆時你的國家包與 playbook 已被兩個市場驗證過，授權才賣得出價

## 7. 不要做的事（東南亞版避坑）

1. **不要五國同時打**——每一國都是一次完整的 GTM，資源分散＝全輸
2. **不要先做支付金流整合**——各國 QR 錢直接進老闆自己的帳（App 只做分類記帳），避開五國支付牌照泥沼；金流是 Stage 3 的 I 模式
3. **不要 fork codebase**——一份代碼、profiles 分市場，否則五國五版本維護到死
4. **不要機翻上線**——收銀術語錯一個字，老闆整天看著它，信任歸零
